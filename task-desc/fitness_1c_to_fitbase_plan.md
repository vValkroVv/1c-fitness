# План обработки выгрузки 1C Fitness в формат Fitbase

Дата подготовки плана: 2026-05-04

## 1. Цель задачи

Нужно подготовить тестовую миграцию клиентов из архива 1C Fitness в Fitbase.

На первом этапе переносится только сегмент **«Действующие клиенты»**. Это клиенты, у которых на дату выгрузки есть открытый/активный абонемент сроком **месяц или более**. Результат нужно выгрузить в два XLSX-файла по уже переданным шаблонам:

1. `Копия Импорт_заявки.xlsx` — основной импорт заявок/клиентов в воронку Fitbase.
2. `Пластиковая карта.xlsx` — отдельная выгрузка телефонов, ФИО и номеров пластиковых карт.

Скрипт нужно делать не как одноразовую ручную обработку, а как воспроизводимый пайплайн: после теста его можно будет переиспользовать для полного боевого переноса и расширить на остальные воронки.

## 2. Исходные данные и ограничения

### 2.1. Что уже есть на сервере

Сервер доступен по SSH:

```text
linuxadmin@192.168.2.36
```

Подключение идет через VPN `fitness`, маршрут через `ppp0`.

Основной файл на сервере:

```text
/home/linuxadmin/Fitnes.bak
```

Параметры файла:

```text
Размер: 12,770,610,688 bytes, примерно 11.9 GiB / 12.8 GB
Права: -rw-rw-r--
Владелец: linuxadmin:linuxadmin
Дата изменения: 2026-04-29 20:57:02 +0000
Тип по file: data
```

RAM достаточно:

```text
Всего RAM: примерно 30 GiB
Доступно RAM: примерно 29 GiB
Swap: 8 GiB
```

Ключевое ограничение — диск:

```text
Корневой filesystem /home: 48G
Занято: 23G
Свободно: 23G
```

В разговоре есть предположение, что восстановленная база может занимать около `100G`. Поэтому нельзя начинать восстановление базы на текущем разделе без проверки фактического размера файлов внутри backup. Нужен отдельный шаг проверки размера и, вероятно, дополнительный диск/том/сервер для восстановления.

### 2.2. Что известно по требованиям заказчика

На тестовом этапе переносим воронку:

```text
Действующие клиенты
```

Критерий попадания в сегмент:

```text
Клиент имеет сейчас открытый/активный абонемент на месяц или более.
```

Этапы внутри воронки:

```text
Бронь
60-31 день до окончания
30-8 дней до окончания
7-0 день до окончания
Действующие
```

Также в `explanation.txt` указан этап:

```text
Разблокирование клиента
```

По нему нет полноценного бизнес-правила в расшифровке разговора, поэтому его нельзя реализовывать молча. Его нужно вынести в список уточнений и подключить только после подтверждения правила.

### 2.3. Шаблон `Копия Импорт_заявки.xlsx`

Лист: `Лист1`.

Файл содержит 9 рабочих колонок. Первая строка — технические имена полей, вторая строка — русские названия, третья строка — пример. Финальная выгрузка должна сохранять структуру шаблона, а данные писать с третьей строки, заменяя пример.

| Колонка | Техническое поле | Название в шаблоне | Что писать |
|---|---|---|---|
| A | `client_id` | `Внутренний номер клиента` | внутренний номер/код клиента из 1C |
| B | `phone` | `Телефон *` | все телефоны клиента через запятую |
| C | `client_fio` | `ФИО клиента *` | ФИО клиента |
| D | `email` | `Почта` | email, если есть |
| E | `funnel` | `Воронка *` | всегда `Действующие клиенты` |
| F | `funnel_step` | `Этап воронки *` | этап по алгоритму |
| G | `budget` | `Бюджет` | пустая ячейка |
| H | `create_date` | `Дата создания *` | первое появление клиента в базе / первая продажа |
| I | `manager` | `Менеджер` | менеджер по правилу распределения |

### 2.4. Шаблон `Пластиковая карта.xlsx`

Лист: `Лист1`.

Файл содержит 3 колонки, данные нужно писать со второй строки.

| Колонка | Название в шаблоне | Что писать |
|---|---|---|
| A | `телефон` | телефоны клиента через запятую |
| B | `фио` | ФИО клиента |
| C | `номер пластиковой карты` | номер пластиковой карты из 1C |

## 3. Общая стратегия

Работу нужно разделить на 8 крупных блоков:

1. Подготовить сервер и рабочую структуру папок.
2. Определить точный формат `Fitnes.bak` и способ доступа к данным.
3. Восстановить базу или получить доступ к данным 1C другим безопасным способом.
4. Найти в базе нужные сущности: клиенты, телефоны, email, абонементы, продажи, брони, карты, точки продаж.
5. Реализовать нормализацию и дедупликацию клиентов.
6. Реализовать бизнес-алгоритм отбора действующих клиентов и этапов воронки.
7. Сформировать два XLSX-файла строго по шаблонам.
8. Проверить результат, отдать на тестовую загрузку в Fitbase, зафиксировать ошибки и доработки.

## 4. Пошаговый план работ

## Шаг 1. Подключиться к серверу и зафиксировать состояние

1. Подключиться через VPN `fitness`.
2. Зайти на сервер:

```bash
ssh linuxadmin@192.168.2.36
```

3. Проверить, что файл на месте:

```bash
ls -lh /home/linuxadmin/Fitnes.bak
du -h /home/linuxadmin/Fitnes.bak
file /home/linuxadmin/Fitnes.bak
```

4. Проверить место на диске:

```bash
df -h
lsblk -f
```

5. Зафиксировать checksum исходного backup, чтобы понимать, что файл не менялся:

```bash
sha256sum /home/linuxadmin/Fitnes.bak > /home/linuxadmin/Fitnes.bak.sha256
```

6. Создать рабочую структуру, не копируя backup внутрь проекта:

```bash
mkdir -p /home/linuxadmin/fitness_migration/{scripts,config,logs,output,tmp,docs}
```

7. Внутри проекта хранить только скрипты, конфиги, логи и результаты. Сам `/home/linuxadmin/Fitnes.bak` использовать по ссылке/пути, без дублирования, потому что свободного места мало.

**Результат шага:** сервер подготовлен, backup не тронут, состояние зафиксировано.

---

## Шаг 2. Проверить формат `Fitnes.bak`

Цель — понять, чем именно является `Fitnes.bak`: backup SQL Server, backup другой СУБД, дамп 1C или иной бинарный формат.

Минимальные проверки:

```bash
file /home/linuxadmin/Fitnes.bak
head -c 4096 /home/linuxadmin/Fitnes.bak | strings | head -100
strings -n 8 /home/linuxadmin/Fitnes.bak | head -200
```

Если признаки указывают на SQL Server backup, нужно поднимать SQL Server или использовать уже доступный SQL Server и выполнить:

```sql
RESTORE HEADERONLY
FROM DISK = '/path/to/Fitnes.bak';

RESTORE FILELISTONLY
FROM DISK = '/path/to/Fitnes.bak';

RESTORE VERIFYONLY
FROM DISK = '/path/to/Fitnes.bak';
```

`RESTORE HEADERONLY` нужен, чтобы посмотреть метаданные backup. `RESTORE FILELISTONLY` нужен, чтобы увидеть логические имена файлов, физические пути и примерный размер MDF/LDF внутри backup. Это главный шаг перед восстановлением, потому что текущие свободные `23G` на `/home` могут быть недостаточны.

Если это не SQL Server backup, нужно определить альтернативный путь:

1. Если это backup PostgreSQL — использовать инструменты PostgreSQL.
2. Если это файловая база 1C — искать/получать `1Cv8.1CD` или выгрузку `.dt`.
3. Если это дамп 1C `.dt`/иной формат — восстанавливать через 1C Configurator.
4. Если формат не определяется — попросить у Андрея/ответственного администратора точное происхождение файла и способ восстановления.

**Результат шага:** понятен тип backup, известен ожидаемый размер восстановленной базы, принято решение, где ее восстанавливать.

## Шаг 3. Решить вопрос с диском до восстановления

Если `RESTORE FILELISTONLY` или другая проверка показывает, что восстановленная база больше доступных `23G`, нужно не пытаться восстановить ее на текущий `/home`.

Варианты решения:

1. Подключить дополнительный диск/раздел к серверу и смонтировать, например, в:

```text
/mnt/fitness_restore
```

2. Увеличить текущий LVM-раздел `/` / `/home`, если администратор может расширить диск.
3. Восстановить backup на другой машине, где уже есть подходящая СУБД и достаточно места.
4. Попросить не `.bak`, а выгрузку из 1C в более удобном формате, если восстановление backup технически нецелесообразно.

Минимальный безопасный запас: восстановленная база + временные файлы + итоговые XLSX + логи. Если ожидаемый размер около `100G`, практически нужен диск больше `120-150G`, а не текущие `23G`.

**Результат шага:** есть среда, где база может быть восстановлена без риска забить корневой раздел.

## Шаг 4. Восстановить базу или получить доступ к данным

### 4.1. Если backup SQL Server

1. Установить/поднять SQL Server в подходящей среде или использовать существующий SQL Server.
2. Скопировать или примонтировать backup так, чтобы SQL Server имел к нему доступ.
3. Выполнить:

```sql
RESTORE FILELISTONLY
FROM DISK = '/path/to/Fitnes.bak';
```

4. Восстановить базу с `WITH MOVE`, чтобы MDF/LDF легли в нужную папку:

```sql
RESTORE DATABASE FitnessRestored
FROM DISK = '/path/to/Fitnes.bak'
WITH
  MOVE 'logical_data_name' TO '/mnt/fitness_restore/FitnessRestored.mdf',
  MOVE 'logical_log_name'  TO '/mnt/fitness_restore/FitnessRestored.ldf',
  RECOVERY,
  STATS = 10;
```

5. Проверить, что база доступна:

```sql
SELECT name, state_desc FROM sys.databases WHERE name = 'FitnessRestored';
```

### 4.2. Если доступ через 1C предпочтительнее прямого SQL

Для 1C прямые SQL-таблицы могут иметь технические имена и зависеть от конфигурации. Поэтому после восстановления лучше сначала понять структуру через 1C/метаданные:

1. Поднять 1C-клиент/конфигуратор или подключиться к восстановленной базе как к информационной базе.
2. Посмотреть, как называются справочники и документы:
   - клиенты/контрагенты/физлица;
   - контактная информация;
   - абонементы/карты/договоры;
   - продажи/чеки/реализации;
   - брони;
   - подразделения/клубы/точки продаж.
3. Если в 1C есть штатные отчеты/обработки/выгрузки, использовать их как источник структуры.
4. Если выгрузка через 1C неудобна, перейти к SQL, но только после сопоставления физических таблиц с бизнес-сущностями.

**Результат шага:** база доступна, можно выполнять запросы и получать бизнес-данные.

## Шаг 5. Найти и описать модель данных

Нужно составить техническую карту, откуда брать каждое поле.

### 5.1. Клиенты

Найти источник:

- внутренний номер клиента;
- ФИО;
- дата создания клиента, если есть;
- связанные телефоны;
- email;
- ссылка на пластиковую карту;
- подразделение/точка приобретения, если хранится.

### 5.2. Телефоны и email

Найти все контактные данные клиента.

Правило для телефонов:

```text
Все телефоны одного клиента объединять в одну ячейку через запятую.
```

Пример:

```text
79999999999, 78888888888
```

Телефоны нужно нормализовать только технически:

- убрать пробелы, скобки, дефисы;
- привести к единому виду, если это безопасно;
- не придумывать номер, если данных нет.

Если телефон отсутствует, клиент попадает в отдельный список ошибок, потому что в шаблоне поле `Телефон *` обязательное.

### 5.3. Абонементы

Найти сущность, где видны:

- клиент;
- тип/номенклатура абонемента;
- дата начала;
- дата окончания;
- статус;
- срок действия;
- блокировка/заморозка/разблокировка, если есть;
- точка продажи/клуб.

Критерий для тестовой выборки:

```text
У клиента есть активный/открытый абонемент на дату выгрузки, и срок абонемента месяц или более.
```

Если у клиента несколько активных абонементов, для расчета этапа брать тот, у которого дата окончания максимальная, если бизнес-логика 1C не подсказывает другое.

### 5.4. Первая дата появления клиента

Для `create_date` нужно брать не дату текущего абонемента, а первое появление клиента в базе.

Приоритет источников:

1. Первая продажа клиента за всю историю.
2. Если продажи нет — дата создания карточки клиента.
3. Если нет обеих дат — вынести клиента в список ошибок/ручной проверки.

### 5.5. Бронь

Нужно найти, как в 1C отмечается бронь:

- отдельный документ;
- статус клиента;
- статус абонемента;
- комментарий/регистр;
- заказ/резерв/предпродажа.

Правило приоритета:

```text
Если у клиента есть актуальная бронь, этап воронки = Бронь, независимо от количества дней до окончания абонемента.
```

Важно: в разговоре звучит формулировка «выделим в отдельную воронку “Бронь”», но по текущему шаблону и списку этапов это нужно реализовать как **этап воронки** `Бронь` внутри воронки `Действующие клиенты`. Если Fitbase требует отдельную воронку, это выяснится на тестовой загрузке.

### 5.6. Пластиковые карты

Найти поле/таблицу с номером пластиковой карты клиента.

Для тестового этапа выгрузку пластиковых карт делать по тому же набору клиентов, который попадает в основной файл `Действующие клиенты`. Если заказчик попросит — расширить на всех клиентов.

### 5.7. Менеджеры

Менеджеров не нужно доставать из 1C. Их нужно проставлять по списку, который должен дать заказчик.

Рекомендуемая схема конфига:

```yaml
branches:
  Гоголевский:
    - Иванов Иван Иванович
    - Петров Петр Петрович
  default:
    - Менеджер 1
    - Менеджер 2
```

Распределение должно быть не настоящим random, а детерминированным, чтобы при повторном запуске один и тот же клиент получал того же менеджера:

```text
manager = managers_for_branch[hash(client_id) % count(managers_for_branch)]
```

Если точка продажи не найдена, использовать `default` или вынести в отчет — в зависимости от требований заказчика.

**Результат шага:** есть таблица соответствия `поле Fitbase -> источник в 1C -> правило обработки`.

## Шаг 6. Зафиксировать бизнес-алгоритм отбора и этапов

### 6.1. Дата расчета

Ввести параметр:

```text
cutoff_date
```

Это дата, относительно которой считаются:

- активность абонемента;
- количество дней до окончания;
- попадание клиента в тестовую воронку.

Для тестовой выгрузки `cutoff_date` можно поставить датой запуска обработки. Для боевого переноса в день X его нужно задавать явно, чтобы результат был воспроизводимым.

### 6.2. Отбор действующих клиентов

Клиент попадает в выгрузку, если выполняются условия:

```text
1. Клиент уникальный после дедупликации.
2. У клиента есть абонемент сроком месяц или более.
3. Абонемент активен/открыт на cutoff_date.
4. У клиента есть телефон и ФИО, иначе Fitbase-импорт может не пройти.
```

Пример логики:

```text
active_subscription =
  subscription.start_date <= cutoff_date
  AND subscription.end_date >= cutoff_date
  AND subscription.duration_days >= 30
  AND subscription.status IN active_statuses
```

Порог `30` дней нужно подтвердить по справочнику абонементов. Если в 1C месячные абонементы считаются 28/29/31 день или определяются по названию номенклатуры, правило нужно адаптировать.

### 6.3. Расчет дней до окончания

```text
days_to_end = subscription.end_date - cutoff_date
```

Считать целыми календарными днями по датам, без учета времени суток.

### 6.4. Приоритет этапов

Порядок важен:

```text
1. Бронь
2. 60-31 день до окончания
3. 30-8 дней до окончания
4. 7-0 день до окончания
5. Действующие
```

Логика:

```text
если has_active_booking:
    funnel_step = "Бронь"
иначе если 31 <= days_to_end <= 60:
    funnel_step = "60-31 день до окончания"
иначе если 8 <= days_to_end <= 30:
    funnel_step = "30-8 дней до окончания"
иначе если 0 <= days_to_end <= 7:
    funnel_step = "7-0 день до окончания"
иначе:
    funnel_step = "Действующие"
```

Клиенты с `days_to_end < 0` не должны попадать в выборку как действующие, если только в 1C нет отдельного статуса, который показывает продление/открытость абонемента несмотря на дату окончания.

### 6.5. `Разблокирование клиента`

Вынести в отдельное уточнение:

```text
Нужно ли использовать этап "Разблокирование клиента"?
Если да — по какому полю/событию в 1C он определяется и какой приоритет имеет относительно "Бронь"?
```

До получения правила этот этап не включать в итоговый алгоритм, чтобы не исказить тестовую загрузку.

**Результат шага:** бизнес-правила описаны в конфиге/документации, их можно проверить на выборке клиентов.

## Шаг 7. Реализовать дедупликацию

Заказчик сказал, что в 1C могут быть дубли клиентов, и из полностью идентичных клиентов нужно оставлять одного.

Рекомендуемая стратегия:

### 7.1. Техническая уникальность

Если в 1C есть один внутренний ID клиента — это первичный идентификатор.

### 7.2. Поиск дублей

Дополнительно искать потенциальные дубли по нормализованным ключам:

```text
normalized_fio + normalized_phone
normalized_fio + email
phone-only, если ФИО совпадает или почти совпадает
```

### 7.3. Что делать с дублями

1. Если записи полностью идентичны — оставить одну.
2. Если записи похожи, но отличаются важными полями — не склеивать молча, а вынести в `duplicates_report`.
3. Если у дублей разные абонементы — выбрать запись с актуальным активным абонементом и максимальной датой окончания, а остальные зафиксировать в логе.
4. Телефоны и email при безопасном объединении можно объединять, чтобы не потерять контактные данные.

### 7.4. Лог дедупликации

Сформировать отдельный отчет:

```text
logs/duplicates_report.csv
```

Поля отчета:

```text
kept_client_id
dropped_client_id
reason
fio
phones
emails
active_subscription_end_date
```

**Результат шага:** в итоговых XLSX один клиент представлен одной строкой, а все спорные случаи видны в отчете.

## Шаг 8. Реализовать скрипт обработки

Скрипт должен быть параметризованным, а не завязанным на текущий тест.

Рекомендуемая структура проекта:

```text
/home/linuxadmin/fitness_migration/
  config/
    settings.yml
    managers.yml
    stage_names.yml
  scripts/
    01_probe_backup.sh
    02_restore_db.sql
    03_schema_discovery.py
    04_extract_clients.py
    05_build_fitbase_xlsx.py
    06_validate_outputs.py
  logs/
  output/
  tmp/
  docs/
```

### 8.1. `settings.yml`

Пример:

```yaml
source:
  backup_path: /home/linuxadmin/Fitnes.bak
  db_type: mssql
  db_name: FitnessRestored

run:
  cutoff_date: "2026-05-04"
  target_funnel: "Действующие клиенты"
  min_subscription_days: 30
  output_prefix: "fitbase_active_clients_test"

output:
  main_template: "/path/to/Копия Импорт_заявки.xlsx"
  card_template: "/path/to/Пластиковая карта.xlsx"
```

### 8.2. `stage_names.yml`

```yaml
funnel: "Действующие клиенты"
stages:
  booking: "Бронь"
  days_60_31: "60-31 день до окончания"
  days_30_8: "30-8 дней до окончания"
  days_7_0: "7-0 день до окончания"
  active: "Действующие"
```

### 8.3. Скрипт извлечения

Скрипт должен получить из восстановленной базы промежуточную нормализованную таблицу:

```text
client_id
fio
phones
email
first_seen_date
active_subscription_id
active_subscription_start_date
active_subscription_end_date
active_subscription_duration_days
days_to_end
has_booking
branch
plastic_card_number
```

Эту таблицу лучше сохранить в промежуточный файл/таблицу:

```text
tmp/active_clients_normalized.csv
```

или во временную SQL-таблицу. Это упростит проверку без постоянного повторного чтения всей базы.

### 8.4. Скрипт генерации XLSX

Скрипт должен:

1. Открыть шаблон `Копия Импорт_заявки.xlsx`.
2. Сохранить первые две строки без изменений.
3. Очистить строку-пример.
4. Записать клиентов с третьей строки.
5. Сохранить файл в `output/`.
6. Открыть шаблон `Пластиковая карта.xlsx`.
7. Сохранить первую строку без изменений.
8. Записать данные со второй строки.
9. Сохранить файл в `output/`.

Имена результатов:

```text
output/fitbase_active_clients_test_import_zayavki_YYYYMMDD.xlsx
output/fitbase_active_clients_test_plastic_cards_YYYYMMDD.xlsx
```

**Результат шага:** есть воспроизводимый скрипт, который из restored DB делает два XLSX.

## Шаг 9. Заполнить основной XLSX

Правила заполнения `Копия Импорт_заявки.xlsx`:

| Поле | Правило |
|---|---|
| `client_id` | внутренний номер клиента из 1C |
| `phone` | все телефоны клиента через запятую |
| `client_fio` | ФИО из карточки клиента |
| `email` | email из контактных данных, если есть |
| `funnel` | всегда `Действующие клиенты` |
| `funnel_step` | по алгоритму этапов |
| `budget` | пустая ячейка |
| `create_date` | первая продажа/первое появление клиента |
| `manager` | по детерминированному распределению из списка менеджеров |

Особые правила:

1. Не добавлять в XLSX лишние колонки.
2. Не переименовывать заголовки.
3. Не менять порядок колонок.
4. `create_date` записывать как настоящую дату Excel, не как произвольную строку.
5. Если Fitbase не принимает пустой `budget`, на тесте проверить вариант `0`, но первичное требование — пустое поле.
6. Если менеджер не может быть назначен, лучше писать пусто и фиксировать в отчете, либо использовать `default`-список — зависит от подтвержденного правила.

## Шаг 10. Заполнить XLSX пластиковых карт

Правила заполнения `Пластиковая карта.xlsx`:

| Поле | Правило |
|---|---|
| `телефон` | все телефоны клиента через запятую |
| `фио` | ФИО клиента |
| `номер пластиковой карты` | номер пластиковой карты из 1C |

Если у клиента нет пластиковой карты:

1. Либо не включать его в файл пластиковых карт.
2. Либо включить с пустой картой.

Рекомендуемый вариант для теста: включать только клиентов, у которых есть номер пластиковой карты, а клиентов без карты вывести в отдельный отчет `missing_cards_report.csv`. Если Fitbase ожидает строки по всем клиентам, это уточнить после тестовой загрузки.

## Шаг 11. Сделать техническую валидацию результата

Перед передачей заказчику обязательно сформировать отчет проверки.

### 11.1. Проверки основного XLSX

1. Заголовки совпадают с шаблоном.
2. Количество строк в XLSX равно количеству уникальных действующих клиентов после фильтров.
3. Нет дублей по `client_id`.
4. Нет дублей по `phone + client_fio`, кроме разрешенных спорных случаев.
5. Нет пустых обязательных полей:
   - `phone`;
   - `client_fio`;
   - `funnel`;
   - `funnel_step`;
   - `create_date`.
6. В `funnel` только значение:

```text
Действующие клиенты
```

7. В `funnel_step` только допустимые значения:

```text
Бронь
60-31 день до окончания
30-8 дней до окончания
7-0 день до окончания
Действующие
```

8. Для клиентов на этапах по дням проверить границы:

```text
60-31: days_to_end от 31 до 60 включительно
30-8:  days_to_end от 8 до 30 включительно
7-0:   days_to_end от 0 до 7 включительно
```

9. Все клиенты с бронью попали в `Бронь`, а не в этап по дням.
10. `budget` пустой.
11. `create_date` не позже даты первой найденной продажи.

### 11.2. Проверки файла пластиковых карт

1. Заголовки совпадают с шаблоном.
2. Телефон и ФИО совпадают с основным файлом.
3. Номера карт не потеряны.
4. Нет дублей по `номер пластиковой карты`, если по бизнес-логике карта должна быть уникальной.
5. Клиенты без карты вынесены в отдельный отчет.

### 11.3. Отчетность

Сформировать:

```text
output/validation_report.md
logs/duplicates_report.csv
logs/missing_required_fields.csv
logs/missing_cards_report.csv
logs/stage_distribution.csv
```

В `stage_distribution.csv` указать количество клиентов по этапам:

```text
funnel_step,count
Бронь,...
60-31 день до окончания,...
30-8 дней до окончания,...
7-0 день до окончания,...
Действующие,...
```

**Результат шага:** перед передачей есть не только XLSX, но и объяснимый контроль качества.

## Шаг 12. Сделать маленькую тестовую выгрузку для Fitbase

Перед полной тестовой выгрузкой лучше собрать мини-набор:

```text
5-10 клиентов с этапом Бронь
5-10 клиентов с этапом 60-31 день до окончания
5-10 клиентов с этапом 30-8 дней до окончания
5-10 клиентов с этапом 7-0 день до окончания
5-10 клиентов с этапом Действующие
несколько клиентов с двумя телефонами
несколько клиентов с email
несколько клиентов с пластиковыми картами
```

Цель мини-теста:

1. Fitbase принимает структуру файла.
2. Fitbase корректно читает первые две строки основного шаблона.
3. Fitbase корректно парсит телефоны через запятую.
4. Fitbase принимает пустой `budget`.
5. Fitbase корректно распознает дату создания.
6. Fitbase принимает названия воронки и этапов ровно в таком виде.
7. Fitbase принимает менеджеров из переданного списка.
8. Отдельная таблица пластиковых карт загружается и связывается как ожидается.

Если Fitbase возвращает ошибки, фиксировать их и менять только то, что нужно для загрузки, не ломая бизнес-логику.

## Шаг 13. Сделать полную тестовую выгрузку действующих клиентов

После мини-теста:

1. Запустить скрипт на всех действующих клиентах.
2. Сформировать два XLSX:

```text
fitbase_active_clients_test_import_zayavki_YYYYMMDD.xlsx
fitbase_active_clients_test_plastic_cards_YYYYMMDD.xlsx
```

3. Сформировать отчеты проверки.
4. Передать заказчику:

```text
1. основной XLSX для импорта заявок;
2. XLSX пластиковых карт;
3. validation_report.md;
4. stage_distribution.csv;
5. duplicates_report.csv, если есть дубли;
6. missing_required_fields.csv, если есть проблемы с обязательными полями.
```

5. Заказчик/Fitbase загружает файлы.
6. Собрать обратную связь:
   - что импортировалось;
   - что не импортировалось;
   - какие поля Fitbase не принял;
   - как отображаются воронки/этапы;
   - корректно ли связались пластиковые карты;
   - корректно ли распределились менеджеры.

## Шаг 14. Подготовить пайплайн к боевому переносу

После успешного теста нужно привести решение к состоянию, пригодному для дня X.

1. Все бизнес-правила вынести в конфиги.
2. Скрипт должен принимать `cutoff_date` параметром.
3. Скрипт должен уметь запускаться повторно без ручной правки кода.
4. Итоговые файлы должны именоваться с датой/временем запуска.
5. Логи должны сохраняться отдельно для каждого запуска.
6. Исходный backup, настройки, версия скрипта и результаты должны архивироваться вместе.
7. Перед боевым переносом нужно взять свежий backup или договориться о времени, когда данные в 1C больше не меняются.
8. Запустить обработку по свежему состоянию базы.
9. Проверить отчеты.
10. Передать финальные файлы в Fitbase.

## 5. Что нужно уточнить до финальной реализации

Эти вопросы не мешают начать техническую часть, но должны быть закрыты до финальной выгрузки:

1. Точный список менеджеров и привязка менеджеров к точкам/филиалам.
2. Что делать с клиентами, у которых нет телефона, если они действующие.
3. Точный порог «абонемент на месяц или более»: 30 дней, календарный месяц или справочник типов абонементов.
4. Что именно в 1C считается «открытым/активным» абонементом.
5. Как в 1C определяется `Бронь`.
6. Нужно ли использовать этап `Разблокирование клиента`; если да — по какому правилу и с каким приоритетом.
7. Нужно ли в файл пластиковых карт включать только клиентов с картами или всех клиентов, оставляя пустой номер карты.
8. Что делать с похожими, но не полностью идентичными дублями.
9. Должен ли `budget` быть строго пустым или Fitbase требует `0`.
10. Должна ли `create_date` быть первой продажей, датой создания карточки клиента или строго конкретным полем 1C, если оно есть.

## 6. Риски и как их закрыть

| Риск | Почему важно | Как закрыть |
|---|---|---|
| Недостаточно диска для восстановления | На сервере свободно около 23G, восстановленная база может быть около 100G | До восстановления выполнить `RESTORE FILELISTONLY`; подключить диск/расширить LVM/восстановить на другой машине |
| Неизвестный формат `.bak` | Нельзя выбрать инструмент восстановления | Проверить сигнатуры и метаданные; при необходимости уточнить у администратора происхождение backup |
| Физические таблицы 1C сложные | Можно неправильно сопоставить бизнес-сущности | Использовать 1C/метаданные/конфигуратор или сначала составить карту таблиц |
| Непонятен статус `Бронь` | Неверное распределение по этапам | Найти поле/документ брони и проверить на примерах |
| Неясен этап `Разблокирование клиента` | Возможна потеря отдельного сегмента | Не реализовывать без подтвержденного правила |
| Дубли клиентов | Fitbase может получить повторные карточки | Дедупликация + отчет по дублям |
| Обязательные поля пустые | Fitbase может отклонить строки | Валидация обязательных полей до передачи |
| Менеджеры не предоставлены | Нельзя заполнить поле `manager` по требованию | Получить список менеджеров и положить в `managers.yml` |
| Fitbase иначе парсит даты/телефоны | Импорт может пройти некорректно | Сначала мини-тест на малой выборке |
